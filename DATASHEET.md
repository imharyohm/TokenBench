## Motivation

**Q1. For what purpose was the dataset created?**

To measure the **token efficiency of codebase-context methods** — RAG, repo-map, graph retrieval, prompt compression, and raw-dump — when answering questions about pinned open-source Python codebases. The benchmark produces an accuracy-vs-tokens Pareto frontier with build-cost amortization at query volumes V ∈ {1, 100, 10000}. It is a **measurement instrument**, not a leaderboard claim: the user reads the curve and decides which method fits their query volume and accuracy regime.

The motivating gap was that public claims of "Nx token efficiency" routinely conflated cold-start vs amortized cost, ignored the priors floor, and used scoring that was vulnerable to harness gaming (UC Berkeley RDI, 2025: 8 major agent benchmarks driven to ~100% by an agent that gamed the harness).

**Q2. Who created the dataset?**

Single author / annotator: GitHub user `hgupta163` (this repository). No outside funding.

**Q3. Who funded the creation?**

Self-funded gateway spend, ~$50 cumulative through v1.0.0.

## Composition

**Q4. What do the instances represent?**

Two task families share `dataset_version: 1.0.0`:

1. **needle-codebase** (300 tasks) — auto-scored "which function is described as: <docstring summary>?" tasks generated by parsing the AST of three pinned Python repositories. Gold answer = the function's short name. Scored by `AutoContainsJudge` (case-insensitive substring match).
2. **swe-qa** (210 tasks) — hand-curated free-form questions about the same three pinned repos. Gold answer = a 3-7 sentence reference text. Scored by `LLMJudge` (`bedrock.anthropic.claude-opus-4-7`, N=3 majority vote, multi-dimensional rubric v1.1.0).

The pinned repos are the dataset's substrate, not its instances. Each task carries a `RepoRef(url, commit, snapshot_sha256)` so the substrate is content-addressable. Repos:

| short_id | commit | license | language |
|---|---|---|---|
| click | `8a1b1a33` | BSD-3-Clause | Python |
| rich | `46cebbb0` | MIT | Python |
| httpx | `b5addb64` | BSD-3-Clause | Python |

**Q5. How many instances are there in total?**

| Family | Total | Public | Held-out (DECISIONS.md #2) |
|---|---:|---:|---:|
| needle-codebase | 300 | 240 | 60 |
| swe-qa | 210 | 168 | 42 |

The held-out split is **never** distributed. Manifests live under `artifacts/_heldout/` which is gitignored.

**Q6. Does the dataset contain all possible instances or is it a sample?**

needle-codebase is a **complete enumeration** of every top-level function/method in the three pinned repos that has a docstring of length ≥30 characters and is not in `_BORING_NAMES` (e.g. `__init__`, `setup`, `main`). It is not a sample.

swe-qa is **hand-curated** by a single annotator. It is a sample of the space of "questions a developer might ask about these libraries"; the population is undefined. The 30-task `sample_chunk6.jsonl` subset used for the rigor sweep is stratified by `(repo, difficulty)` via largest-remainder allocation (`scripts/sample_swe_qa.py`, seed=1).

**Q7. What data does each instance consist of?**

Schema in `tokenbench/core/schemas.py:Task`. Required fields:

```
task_id          str   globally unique, e.g. "needle-click-0000" or "swe-rich-0042"
dataset_version  str   "1.0.0"
task_type        str   "needle_function" | "repo_qa"
question         str   prompt text passed to the answering model
repo             RepoRef  (url, commit, snapshot_sha256)
gold             str   reference answer — NEVER passed to the answering provider
needle           str|None  short symbol for needle-codebase; None for swe-qa
scoring          str   "auto_contains" | "llm_judge"
canary           str   "TOKENBENCH-CANARY-…"
license          str   inherited from the pinned repo
meta             dict  diagnostic info (file path, qualified symbol, etc.)
```

**Q8. Is there a label or target associated with each instance?**

Yes: `gold` (and for needle, `needle`). For SWE-QA, 210 human pass/fail labels are committed at `artifacts/swe_qa/v1.0.0/human_labels.jsonl` (Chunk 5 deliverable 3, single annotator, 20-task test-retest sub-sample with ≥48 hr gap).

**Q9. Is any information missing from individual instances?**

`docker_image` on `RepoRef` is `None` until `scripts/build_images.py` produces digest receipts. This is intentional: the v1.0 reproducibility path does not require Docker.

**Q10. Are relationships between individual instances made explicit?**

No. Tasks are independent.

**Q11. Are there recommended data splits?**

Yes — DECISIONS.md #2: 80% public / 20% held-out, stratified by (language, repo) for needle and (repo, difficulty) for swe-qa. Tasks are assigned at `dataset_version` freeze and never re-assigned (DECISIONS.md #7 covers rotation). Manifests:

- `artifacts/needle/v1.0.0/public_split.tsv` (tracked)
- `artifacts/swe_qa/v1.0.0/public_split.tsv` (tracked)
- `artifacts/_heldout/needle/v1.0.0/heldout_split.tsv` (gitignored)
- `artifacts/_heldout/swe_qa/v1.0.0/heldout_split.tsv` (gitignored)

**Q12. Are there any errors, sources of noise, or redundancies?**

Three are documented and load-bearing for interpreting results:

1. **`auto_contains` short-needle leniency.** Symbols like `meta` or `print_calendar` will match any prose containing the word, regardless of intent. The `ExploitBaselineProvider` exposes a +0.21 paired uplift over zero-context on the needle dataset purely from output-side priming + this leniency, with no gold leakage (`task.gold`/`task.needle` reads are absent — verified by `tests/test_baselines.py`). Tightening the scorer to whole-token match would be a `dataset_version` bump (DECISIONS.md #6 forbids tuning published configs); the right home for this caveat is here. See `research/exit_gate_2_priors_floor.md` for the diagnosis and the reframed exit gate (paired uplift CI vs zero-context, T=0.20).
2. **Frontier model priors on public Python repos.** Zero-context (priors-only, empty `<context>`) reaches acc=0.26 on needle/sonnet-4-5 because the model has read public click/rich/httpx commits in pretraining. Real providers' uplift over the priors floor is the meaningful headline number; raw accuracy alone is not.
3. **SWE-QA single-annotator labels.** The 210 human pass/fail labels were authored by `hgupta163` alone with a 20-task test-retest sub-sample ≥48 hr apart. Inter-annotator agreement was not measured (no second annotator). The judge-vs-human calibration κ=0.806 is *human-vs-judge* agreement, not human-vs-human reliability.

**Q13. Is the dataset self-contained or does it link to external resources?**

Self-contained for the question/gold/manifest data, which is stored in this repository. The pinned repo source trees are NOT stored; they are reproduced on first run by `scripts/snapshot_repos.py`, which clones at the pinned commit and verifies the resulting tree's `sha256` against `repo_pins.py`. If the upstream repo is deleted, content-addressed fallbacks would need to be added.

**Q14. Does the dataset contain confidential, offensive, or personal data?**

No. All content derives from public open-source Python libraries (click, rich, httpx) under permissive licenses.

## Collection Process

**Q15. How was the data acquired?**

- needle-codebase: Python AST traversal of pinned repo source trees, filtered to functions/methods with docstrings ≥30 chars. Generated by `tokenbench/datasets/needle_codebase.py` at load time — there is no static questions file for needle.
- swe-qa: hand-authored via an interactive workflow (`scripts/label_swe_qa.py`), one question + reference answer at a time. The annotator inspected the live source at the pinned commit while writing the question.

**Q16. What mechanisms or procedures were used to collect the data?**

Code-only. No crowd workers, no APIs (other than `git clone` to fetch the pinned repos and the gateway calls used to score, which are not collection).

**Q17. If the dataset is a sample, what was the sampling strategy?**

The `sample_chunk6.jsonl` rigor-sweep subsample uses largest-remainder allocation per `(repo, difficulty)` stratum, seeded at 1. See `scripts/sample_swe_qa.py`.

**Q18. Who was involved in the data collection process?**

Single author `hgupta163`. No outside annotators, no funding bodies.

**Q19. Over what timeframe was the data collected?**

2026-06-09 through 2026-06-13. Pre-registration boundary at git commit `7438ee7`; human labels added at `d95de96`; rubric v1.1.0 calibration locked at `757d65b`. Verifiable via `git log`.

**Q20. Were any ethical review processes conducted?**

No formal IRB. The annotator labels their own dataset, with no human subjects beyond themselves.

## Preprocessing / cleaning / labeling

**Q21. Was any preprocessing/cleaning/labeling of the data done?**

- needle: docstring sentences are extracted and lightly normalized (whitespace collapse, first-sentence split on `. ` / `.\n`).
- swe-qa: questions and reference answers are stored verbatim as authored.
- LLM judge candidate text is normalized only by stripping leading/trailing whitespace before the judge sees it (no content rewriting).

**Q22. Was the "raw" data saved in addition to the preprocessed data?**

Pinned repo source trees (raw) are reconstructible from `(url, commit)` + `snapshot_sha256` verification.

**Q23. Is the software for preprocessing available?**

Yes, in this repository under `tokenbench/` and `scripts/`. Pinned dependencies in `repro/requirements.lock.txt` (103 packages frozen at end of Chunk 4).

## Uses

**Q24. Has the dataset been used for any tasks already?**

The Chunk 3 sweep (5 providers × 2 models × 24 needle tasks × 2 repeats, 243 records) and the Chunk 6 rigor sweep (scope F-trim — see `run_chunk6_rigor.py` and `results/runs/chunk6_rigor.jsonl`) used the dataset. SWE-QA judge calibration on the full 210-task gold set produced κ=0.806, ECE=0.092 under rubric v1.1.0 (`results/judge/calibration_judge-5f2c4466_rubric1.1.0.json`).

**Q25. Is there a repository that links to any or all papers or systems that use the dataset?**

Not yet. v1.0.0 is local-only; the project plans to push to GitHub from a second device after v1.0 is hardened (locked setting #6 in CONTEXT_HANDOFF.md).

**Q26. What other tasks could the dataset be used for?**

Suitable for studies of:
- Retrieval/compression token efficiency under realistic answer-quality constraints.
- LLM-judge calibration on free-form code QA with separated answering vs grading models.
- Contamination defense — the held-out split is reserved for this.
- Probing model priors on public OSS Python codebases (zero-context as a baseline).

NOT suitable as a model-quality benchmark on its own — the answering-model dimension is small (2 models × 5 providers in this release), and the SWE-QA reference answers are a single annotator's interpretation.

**Q27. Are there tasks for which the dataset should not be used?**

- **Multi-language code retrieval claims.** All three pinned repos are Python. Generalizing TokenBench numbers to TypeScript/Java/Rust/etc is out of scope.
- **Long-horizon agent planning.** Tasks are single-question, not multi-turn. The agentic provider (`SweBenchProProvider`) is deferred to v1.1; v1.0 measures static context methods only.
- **Cherry-picked headline ratios** like "method X is 71.5× more efficient." The benchmark deliberately publishes the curve; pulling a single ratio out of context defeats the design.

**Q36. Is there an erratum?**

**Q37. Will the dataset be updated?**

Yes, on the schedule pinned in DECISIONS.md #7:

- **Cadence:** rotate the held-out split every 12 months by default.
- **Early-rotation triggers:**
  - public-vs-held-out gap exceeds 2× bootstrap CI for any submitted method (suspected contamination)
  - any task in the held-out split is publicly leaked
  - canary string for the dataset version appears in any model's training corpus disclosure
- **Past leaderboard entries on rotation:** archived under `dataset_version: <old>` with a `superseded_by: <new>` pointer; old entries are never deleted but are not displayed on the active leaderboard.

**Q38. If the dataset relates to people, are there applicable retention limits?**

N/A.

**Q39. Will older versions of the dataset continue to be supported / hosted / maintained?**

Yes. Run records are versioned by `(dataset_version, harness_version, provider_version, model)` and are append-only (DECISIONS.md #4). An old entry is never overwritten.

**Q40. If others want to extend / augment / build on the dataset, is there a mechanism for them to do so?**

Yes — the submission protocol lives in `LEADERBOARD.md`. Submitters provide a Provider plugin + frozen config; the harness scores it on both public and held-out splits, and the gap is reported on the leaderboard alongside the headline TPCA(V) curve.

## Calibration history (project-specific addendum)

The judge rubric was bumped 1.0.0 → 1.1.0 on 2026-06-13 after per-dimension diagnostics on the saved 210-task vote dims showed the `faithfulness` dimension failed to discriminate (mean 1.73 on human-pass vs 1.66 on human-fail; gap 0.07 vs ~1.0 on the other dims). Lowering its pass floor from `==2` to `≥1` raised κ from 0.609 to 0.806 and dropped ECE from 0.20 to 0.09. The rubric prompt was unchanged; only the binary aggregation rule moved. Both v1.0.0 and v1.1.0 calibration numbers are retained for transparency:

- `results/judge/calibration_judge-5f2c4466.json` (v1.0.0; FAIL: ECE 0.20 > 0.10)
- `results/judge/calibration_judge-5f2c4466_rubric1.1.0.json` (v1.1.0; PASS: κ=0.806, ECE=0.092)

Detail in DECISIONS.md change log entry dated 2026-06-13.

## Frozen configs (DECISIONS.md #6)

The 5 v1.0 providers (`rag-bm25`, `repo-map`, `graphify`, `llmlingua-rag`, `raw-dump`) ship with frozen published configs. Tuning a config to inflate accuracy on this dataset is forbidden by design: switching to equal-budget evaluation would require a new `dataset_version`. This datasheet is the right place to document that constraint.
